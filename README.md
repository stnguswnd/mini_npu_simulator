# Mini NPU Simulator

과제 요구사항에 맞춰 구현한 Python 콘솔 프로그램입니다.

입력 패턴이 `Cross` 형태에 더 가까운지, `X` 형태에 더 가까운지를 **MAC(Multiply-Accumulate)** 방식으로 계산해 판정합니다.

구현 범위는 아래를 모두 포함합니다.

- 모드 1: 사용자 3x3 입력 분석
- 모드 2: `data.json` 일괄 분석
- 라벨 정규화 (`+`, `cross`, `x` → `Cross`, `X`)
- epsilon 기반 동점 처리
- 크기별 성능 분석
- 보너스 1: 2차원 배열 → 1차원 배열(flat) 최적화 비교
- 보너스 2: Cross / X 패턴 자동 생성기

외부 라이브러리는 사용하지 않았고, `json`, `time`, `argparse`, `pathlib`, `re` 등 표준 라이브러리만 사용했습니다.

---

## 1. 프로젝트 개요

이 프로그램의 핵심 목적은 입력 패턴과 필터를 위치별로 곱하고, 그 결과를 모두 누적해 점수를 만든 뒤,

어느 필터와 더 유사한지를 판정하는 것입니다.

전체 흐름은 아래와 같습니다.

1. 입력을 받는다.
2. 입력 형식과 크기를 검증한다.
3. 라벨을 내부 표준 형식으로 정규화한다.
4. 패턴 크기에 맞는 필터를 찾는다.
5. MAC 연산으로 `Cross` 점수와 `X` 점수를 계산한다.
6. 두 점수를 비교해 최종 결과를 판정한다.
7. 결과를 출력하거나 `expected`와 비교해 PASS/FAIL을 판단한다.

즉, 이 과제는 단순히 결과만 출력하는 문제가 아니라,

**입력 처리 → 정규화 → 크기 매칭 → MAC 계산 → 판정 → 출력**까지 전체 구조를 안정적으로 설계하는 것이 중요합니다.

```bash

```



## 4. `data.json` 구조

프로그램은 아래 구조를 기준으로 JSON을 읽습니다.

```json
{
  "filters": {
    "size_5": {
      "cross": [[0, 0, 1, 0, 0], "..."],
      "x": [[1, 0, 0, 0, 1], "..."]
    },
    "size_13": {
      "cross": [[0, 0, 0, "..."], "..."],
      "x": [[1, 0, 0, "..."], "..."]
    },
    "size_25": {
      "cross": [[0, 0, 0, "..."], "..."],
      "x": [[1, 0, 0, "..."], "..."]
    }
  },
  "patterns": {
    "size_5_1": {
      "input": [[1, 0, 0, 0, 1], "..."],
      "expected": "x"
    },
    "size_13_1": {
      "input": [[0.5, 0.0, 0.0, "..."], "..."],
      "expected": "+"
    }
  }
}
```

### 내부 표준 라벨

프로그램 내부에서는 아래 두 라벨만 표준으로 사용합니다.

- `Cross`
- `X`

입력 데이터에 다양한 표기가 들어와도, 비교 전에 모두 표준 라벨로 변환합니다.

- `+` → `Cross`
- `cross` → `Cross`
- `x` → `X`

이렇게 하면 이후 판정 로직은 항상 동일한 기준으로 동작할 수 있습니다.

## 5. 코드 구조와 설계

## 5-1. 전체 구조

프로그램은 크게 다음 단계로 동작합니다.

### 모드 1: 사용자 입력 분석

- 사용자가 3x3 입력을 직접 입력
- 입력 형식 검증
- `Cross` / `X` 점수 계산
- 모드 1 규칙에 맞게 결과 출력

### 모드 2: JSON 일괄 분석

- `data.json` 로드
- `filters`, `patterns` 구조 검증
- 패턴 크기에 맞는 필터 매칭
- 각 패턴별 MAC 계산
- `expected`와 비교해 PASS/FAIL 출력

즉, 코드 구조는 단순 계산 함수 하나가 아니라

**입력 처리 계층, 데이터 정규화 계층, 연산 계층, 판정 계층, 출력 계층**으로 나누어 생각할 수 있게 설계했습니다.

---

## 5-2. MAC 연산 로직의 전체 흐름

MAC 연산은 `mac_score()` 함수에서 수행합니다.

동작 순서는 아래와 같습니다.

1. 입력 행렬과 필터 행렬의 각 위치를 순회한다.
2. 같은 위치의 값을 곱한다.
3. 곱한 결과를 누산 변수에 더한다.
4. 모든 위치 계산이 끝나면 누적 합을 점수로 반환한다.

즉, 흐름을 코드 구조 기준으로 정리하면 다음과 같습니다.

**입력 행렬 준비 → 위치별 곱셈 → 누적 합산 → 최종 점수 반환**

예를 들어 3x3 입력이면 총 9개의 위치를 방문하고, 각 위치마다 곱셈과 덧셈을 수행합니다.

이 방식은 NPU의 기본 MAC 동작을 가장 단순하게 파이썬 코드로 옮긴 형태입니다.

## 5-3. `patterns` 키에서 크기 N을 추출하고 `filters`의 `size_N`과 매칭한 방식

`patterns`에는 `size_5_1`, `size_13_2`처럼 패턴 크기와 인덱스가 함께 들어 있습니다.

여기서 실제로 필요한 것은 “이 패턴이 몇 x 몇 크기인가” 입니다.

그래서 아래 순서로 설계했습니다.

1. 패턴 키에서 `N` 값을 추출한다.
2. 예를 들어 `size_5_1`이면 `N=5`를 얻는다.
3. `filters`에서 `size_5`를 찾는다.
4. 같은 크기의 `Cross`, `X` 필터와 입력 패턴을 비교한다.

예:

- `size_5_1` → `filters["size_5"]`
- `size_13_4` → `filters["size_13"]`

이 설계의 장점은 다음과 같습니다.

- 패턴 수가 늘어나도 분기문이 복잡해지지 않음
- 크기별 필터를 재사용할 수 있음
- 새로운 테스트 케이스 추가가 쉬움
- 잘못된 키 형식도 초기에 검증 가능함

즉, 패턴 이름과 필터 이름을 하드코딩으로 일일이 연결한 것이 아니라,

**크기 정보 기반으로 자동 매칭되도록 설계**한 것입니다.

---

## 5-4. `normalize_label()`을 별도 함수로 분리한 이유와 장점

입력 데이터에는 다음처럼 여러 라벨 표기가 섞여 들어올 수 있습니다.

- `+`
- `cross`
- `Cross`
- `x`
- `X`

이 값을 판정 로직 안에서 매번 직접 처리하면 코드가 복잡해지고,

비교 누락이나 오타로 인한 버그가 생기기 쉽습니다.

그래서 라벨 정규화를 별도 함수로 분리했습니다.

장점은 아래와 같습니다.

1. **단일 책임 원칙**
    - 라벨 해석 책임을 한 곳에 모을 수 있음
2. **판정 로직 단순화**
    - 이후 로직은 `Cross`, `X` 두 값만 기준으로 동작
3. **확장성**
    - 새로운 라벨이 추가되면 정규화 함수만 수정하면 됨
4. **재사용성**
    - `expected`, 필터 이름, 출력 전처리에 공통 사용 가능

즉, 정규화 함수는 단순 편의 기능이 아니라

**입력 표현과 내부 로직을 분리하는 설계 요소**입니다.

## 5-5. 시간 측정 시 I/O를 배제하고 연산 구간만 측정한 기준

성능 측정에서는 파일 읽기, JSON 파싱, 콘솔 출력 같은 I/O 시간을 제외하고

**순수 연산 구간만 측정**하도록 경계를 잡았습니다.

측정 경계는 아래와 같습니다.

- 시작: 입력 행렬과 필터가 모두 준비된 직후
- 종료: MAC 점수 계산이 끝난 직후

즉, 아래 항목은 시간 측정에서 제외했습니다.

- 파일 열기/읽기
- JSON 파싱
- 사용자 입력
- 안내 문구 출력
- 결과 리포트 출력

이렇게 한 이유는 과제에서 비교하고 싶은 대상이

“입출력 속도”가 아니라 **MAC 연산 자체의 성능**이기 때문입니다.

## 6. 핵심 기술 원리

## 6-1. MAC 점수가 높을수록 더 유사하다고 판단할 수 있는 이유

MAC는 입력값과 필터값을 같은 위치끼리 곱해서 모두 더하는 방식입니다.

즉,

- 입력 패턴이 특정 필터의 중요한 위치와 잘 맞으면 큰 값이 많이 누적되고
- 잘 맞지 않으면 누적 합이 작아집니다

예를 들어 `Cross` 필터의 중심 행/열이 강조되어 있는데, 입력도 같은 위치에서 큰 값을 가지면

`Cross` 점수는 커지고, 반대로 `X` 필터의 대각선과는 덜 맞으면 `X` 점수는 작아집니다.

그래서 두 필터 중 더 큰 MAC 점수를 얻은 쪽을

입력과 더 유사한 패턴이라고 판단할 수 있습니다.

---

## 6-2. 패턴 크기가 N일 때 연산 횟수가 N²로 증가하는 이유

입력이 N x N 행렬이면 전체 원소 수는 N²개입니다.

MAC 연산은 모든 위치를 한 번씩 방문하므로:

- 곱셈 횟수: N²
- 덧셈 횟수: N²에 비례

예를 들어:

- 3x3 → 9개
- 5x5 → 25개
- 13x13 → 169개
- 25x25 → 625개

따라서 전체 실행 시간은 입력 크기에 대해 `O(N²)` 경향을 보입니다.

실제 측정 결과에서도 크기가 커질수록 시간이 증가하며, 이는 모든 원소를 직접 순회하는 구조와 일치합니다.

---

## 6-3. 부동소수점 오차와 epsilon 비교 정책이 필요한 이유

실수는 컴퓨터 내부에서 항상 정확히 저장되지 않고, 근사값으로 표현되는 경우가 있습니다.

예를 들어 사람이 보기엔 같은 값이어도 내부적으로는

- `0.9`
- `0.899999999999`

처럼 다르게 저장될 수 있습니다.

그래서 단순히 `score_cross == score_x`로 비교하면,

사실상 같은 값도 다르다고 판정할 위험이 있습니다.

이를 막기 위해 epsilon을 기준으로 비교합니다.

```python
abs(score_cross - score_x) < epsilon
```

즉, “정확히 같은가”가 아니라

“충분히 가까운가”를 기준으로 동점 여부를 판단한 것입니다.

---

## 6-4. 동점 처리 규칙을 mode1과 mode2에서 다르게 둔 이유

동점 판정 기준 자체는 두 모드에서 동일합니다.

둘 다 두 점수 차이가 epsilon보다 작으면 동점으로 간주합니다. 과제 문서에서도 점수 비교는 허용오차(epsilon) 기반으로 수행하고, 예시 기준으로 `abs(score_a - score_b) < 1e-9`이면 동점으로 본다고 안내합니다.

다만 두 모드는 **출력 목적**이 다릅니다.

- **mode1(사용자 입력 3x3)** 은 사용자가 직접 입력한 필터 A와 필터 B를 비교해 보는 모드입니다. 과제 문서에서도 mode1의 출력은 두 필터의 MAC 점수, 연산 시간, 판정 결과를 `A/B/판정 불가` 형태로 보여주도록 설명하고 있습니다. 즉 이 모드에서는 “표준 라벨”보다 **사용자가 입력한 첫 번째 필터와 두 번째 필터 중 어느 쪽이 더 높은가**를 직관적으로 보여주는 것이 목적입니다.
- **mode2(data.json 분석)** 는 JSON에 들어 있는 패턴을 표준 라벨 `Cross`, `X` 기준으로 판정하고, 그 결과를 `expected`와 비교해 PASS/FAIL을 내는 검증 모드입니다. 과제 문서에서도 mode2는 케이스별 판정값을 `Cross/X/UNDECIDED`로 출력하고, `expected`와 같으면 PASS, 다르면 FAIL로 처리하도록 명시하고 있습니다.

즉, **동점 처리 규칙이 다른 것이 아니라**,

같은 동점 상황을 각 모드의 목적에 맞게 다르게 표현한 것입니다.

### mode1의 흐름

1. 사용자가 필터 A, 필터 B, 입력 패턴을 넣는다.
2. A 점수와 B 점수를 계산한다.
3. 점수 차이가 epsilon보다 작으면 `판정 불가`로 출력한다.
4. 그렇지 않으면 더 큰 점수를 가진 쪽을 `A` 또는 `B`로 출력한다.

예를 들어:

- A 점수 = 1.0
- B 점수 = 5.0
→ 판정: `B`

또는

- A 점수 = 0.9000000000
- B 점수 = 0.8999999999
→ 두 점수 차이가 epsilon보다 작다면 판정: `판정 불가`

이 모드는 사용자가 직접 비교 실험을 해보는 용도이므로, 별도의 `expected` 정답값과 비교하는 PASS/FAIL 단계는 없습니다. 과제 예시에서도 mode1은 점수와 판정 결과를 보여주는 데 초점이 있습니다.

### mode2의 흐름

1. `data.json`에서 패턴과 필터를 읽는다.
2. 입력 라벨과 필터 키를 내부 표준 라벨 `Cross`, `X`로 정규화한다.
3. `Cross` 점수와 `X` 점수를 계산한다.
4. 점수 차이가 epsilon보다 작으면 `UNDECIDED`로 판정한다.
5. 판정값과 `expected`를 비교한다.
6. 같으면 PASS, 다르면 FAIL을 출력한다. 과제 문서의 출력 규칙도 동일하다.

### PASS/FAIL 예시

### 예시 1: 정상 일치 케이스

- Cross 점수: 5.0
- X 점수: 1.0
- 판정: `Cross`
- expected: `Cross`

이 경우 판정값과 expected가 같으므로:

```
판정: Cross | expected: Cross | PASS
```

### 예시 2: 반대 패턴 일치 케이스

- Cross 점수: 1.0
- X 점수: 5.0
- 판정: `X`
- expected: `X`

이 경우도 같으므로:

```
판정: X | expected: X | PASS
```

### 예시 3: 동점 처리로 인한 FAIL

- Cross 점수: 0.9000000000000000
- X 점수: 0.8999999999999999
- 점수 차이 < epsilon
- 판정: `UNDECIDED`
- expected: `X`

이 경우 과제 예시처럼 동점 규칙에 따라 `UNDECIDED`가 나오고, 이는 expected인 `X`와 다르므로:

```
판정: UNDECIDED | expected: X | FAIL
사유: 동점(UNDECIDED) 처리 규칙에 따라 FAIL
```

과제 PDF의 data.json 분석 예시에서도 실제로 `size_13_1` 케이스가 이런 형태로 제시되어 있고, `UNDECIDED | expected: X | FAIL (동점 규칙)`으로 설명되어 있다.

### 정리

mode1은 **사용자 입력 필터 A/B를 직관적으로 비교해 주는 실험용 출력**이 목적이고,

mode2는 **표준 라벨 Cross/X 기준으로 expected와 비교해 PASS/FAIL까지 검증하는 채점용 출력**이 목적입니다.

따라서 동점 판정 기준은 같지만, mode1은 `판정 불가`, mode2는 `UNDECIDED`로 표현하고,

특히 mode2에서는 그 다음 단계로 `predicted == expected` 여부에 따라 PASS/FAIL이 이어지도록 설계했습니다.

## 8. 보너스 구현

## 8-1. 보너스 1: 2차원 배열 → 1차원 배열(flat) 최적화

기본 구현은 2차원 배열 인덱싱(`matrix[row][col]`)을 사용합니다.

최적화 버전은 행렬을 1차원 배열로 펴서 선형 접근합니다.

두 버전 모두 전체 원소를 모두 방문하므로 시간 복잡도는 동일합니다.

- 기본 버전: `O(N²)`
- flat 버전: `O(N²)`

하지만 flat 방식은 다음 이점이 있습니다.

- 인덱싱 단계가 더 단순함
- 파이썬 레벨 인덱싱 오버헤드를 줄일 수 있음
- 메모리 접근 패턴이 더 단순함

즉, 이 최적화는 빅오를 바꾸는 것이 아니라

**상수 계수를 줄이는 최적화**입니다.

---

## 8-2. 보너스 2: 패턴 생성기

`generate_cross_pattern(size)`와 `generate_x_pattern(size)`를 구현했습니다.

이 함수들은 아래 기능에서 재사용됩니다.

- `-generate N` 출력
- 성능 분석용 입력 생성
- 예시 데이터 생성

즉, 패턴 생성 기능을 단순 출력용이 아니라

다른 기능에서도 재사용 가능한 유틸리티로 설계했습니다.

## 9. 결과 리포트

예시 `data.json`에는 의도적으로 동점 성격의 테스트를 포함할 수 있습니다.

예를 들어 어떤 입력이 `Cross`와 `X`를 비슷한 비율로 섞은 형태라면,

두 점수가 거의 같아져 동점 처리될 수 있습니다.

이 경우 프로그램은 epsilon 정책에 따라 `UNDECIDED`를 반환합니다.

만약 `expected`가 `X`였다면 최종 결과는 FAIL이 됩니다.

이것은 구현 오류가 아니라,

**동점 처리 규칙이 설계대로 동작한 결과**입니다.

## 10. 시간 복잡도와 성능 해석

MAC 연산은 입력 패턴의 모든 칸을 한 번씩 방문하므로 시간 복잡도는 `O(N²)`입니다.

예를 들어:

- 3x3: 9회
- 5x5: 25회
- 13x13: 169회
- 25x25: 625회

최적화 버전 역시 모든 원소를 순회하므로 점근적 복잡도는 동일합니다.

다만 실제 실행 시간은 flat 버전이 더 유리할 수 있습니다.

즉, 이 과제에서 성능 최적화는

**복잡도 자체를 바꾸는 것이 아니라, 같은 복잡도 안에서 실제 실행 시간을 줄이는 방향**으로 접근했습니다.

### 예시 성능 측정 (`-repeat 1000`)

| 크기 | 기본 MAC 평균(ms) | 최적화 MAC 평균(ms) | 개선율 | 연산 횟수 |
| --- | --- | --- | --- | --- |
| 3x3 | 0.000573 | 0.000280 | 51.03% | 9 |
| 5x5 | 0.001073 | 0.000654 | 39.08% | 25 |
| 13x13 | 0.005692 | 0.003526 | 38.06% | 169 |
| 25x25 | 0.020286 | 0.015492 | 23.63% | 625 |

이 표에서 확인할 수 있듯이, 연산 횟수 `N²`가 증가할수록 실행 시간도 증가합니다.

또한 flat 버전은 모든 크기에서 기본 버전보다 더 낮은 시간을 보여,

선형 접근 최적화가 실제로 도움이 되었음을 확인할 수 있습니다.

## 11-2. 1000x1000 패턴을 처리할 때의 문제와 우선 적용할 최적화

1000x1000 패턴은 총 1,000,000개의 원소를 가지므로, 현재처럼 순수 Python 이중 반복문으로 처리하면 시간과 메모리 부담이 크게 증가합니다.

### 시간 측면

- 필터 하나와 비교할 때도 백만 개 위치에 대해 곱셈-누산이 필요합니다.
- `Cross`, `X`처럼 여러 필터를 비교하면 연산량이 그만큼 추가됩니다.
- 순수 Python 반복문은 인터프리터 오버헤드가 커서, 입력 크기가 커질수록 실행 시간이 빠르게 증가합니다.

### 메모리 측면

- 1000x1000 크기의 2차원 리스트는 원소 수 자체도 많지만, 파이썬 리스트 구조의 오버헤드까지 포함되어 메모리 사용량이 커집니다.
- 필터, 입력, flat 배열, 중간 결과 등을 동시에 유지하면 메모리 부담이 더 커질 수 있습니다.

### 현재 구현 기준에서 우선 적용할 최적화

현재 과제 범위와 구현 방식 안에서 가장 먼저 적용할 수 있는 최적화는 다음과 같습니다.

1. **2차원 배열을 1차원 배열로 평탄화**
    - 인덱싱을 단순화해 접근 오버헤드를 줄일 수 있습니다.
2. **반복문 내부 분기 최소화**
    - 반복문 안에서 불필요한 조건문이나 함수 호출을 줄여 상수 시간을 줄일 수 있습니다.
3. **불필요한 중간 객체 생성 최소화**
    - 임시 리스트나 복사 연산을 줄여 메모리 사용량과 추가 오버헤드를 줄일 수 있습니다.

### 더 큰 규모에서 고려할 수 있는 확장 방향

과제 범위를 넘어 더 큰 입력을 안정적으로 처리해야 한다면, 다음과 같은 방법을 고려할 수 있습니다.

1. **NumPy 벡터화**
    - Python 반복문 대신 C 레벨 연산으로 처리하므로 대규모 배열 연산에서 훨씬 효율적입니다.
    - 예를 들어 원소별 곱셈과 합산을 벡터 연산으로 한 번에 수행할 수 있습니다.
2. **병렬 처리**
    - 필터별 계산이나 큰 배열 블록 단위 계산을 병렬화할 수 있습니다.
    - 다만 Python에서는 GIL 영향이 있어 멀티스레드보다 멀티프로세싱이나 외부 라이브러리 활용이 더 현실적입니다.
3. **GPU / NPU 같은 하드웨어 가속**
    - MAC처럼 동일한 연산을 대량 반복하는 작업은 병렬 연산 장치에서 훨씬 유리합니다.
    - 입력이 클수록 CPU 직렬 처리보다 GPU/NPU 가속의 이점이 커집니다.

즉, 이번 과제 구현에서는 **flat 배열과 반복문 단순화가 가장 현실적인 1차 최적화**이고,

## 입력 크기가 더 커지는 상황에서는 **NumPy 벡터화, 병렬화, 하드웨어 가속**까지 확장 방향으로 설명할 수 있습니다.

## 11-3. CPU의 직렬 처리와 NPU의 병렬 처리 관점에서 병목은 어디인가

이 문제의 핵심 병목은 **대량의 곱셈-누산 반복**입니다.

CPU에서 순수 파이썬 반복문으로 처리하면:

- 원소를 거의 순차적으로 하나씩 계산
- 반복 횟수가 많아질수록 인터프리터 오버헤드가 커짐
- 큰 입력에서 병목이 쉽게 발생

반면 NPU와 같은 병렬 처리 장치는:

- 많은 MAC 연산을 동시에 처리 가능
- 동일한 연산을 대량 반복하는 작업에 강함
- 데이터가 클수록 병렬 처리 이점이 커짐

즉, 이 문제에서는 **원소 단위 곱셈-누산 반복 자체가 병목**이며,

CPU에서는 직렬 반복 비용이, NPU에서는 병렬 MAC 유닛 활용과 메모리 공급 효율이 중요한 포인트가 됩니다.
